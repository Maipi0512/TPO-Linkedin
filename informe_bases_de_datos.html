LinkPro

Arquitectura de bases de datos del proyecto

Ingeniería de Datos II · Trabajo Práctico Obligatorio · 2026

LinkPro es una red social profesional inspirada en LinkedIn. Permite a los usuarios gestionar su perfil, publicar y buscar empleos, conectarse con otros profesionales, compartir contenido en un feed, intercambiar mensajes y recibir notificaciones.

El sistema aplica el patrón de persistencia políglota: en lugar de forzar todos los datos en un único motor, usamos cuatro bases de datos distintas — cada una elegida según el tipo de dato que maneja y el patrón de acceso que requiere. Grandes plataformas como LinkedIn, Netflix y Facebook usan exactamente esta estrategia.

Base de datos 1

Relacional · SQL Supabase — PostgreSQL

¿Qué datos guarda?

usuarios empresas ofertas de trabajo postulaciones historial de estados habilidades educación experiencia laboral grupos miembros de grupos

¿Para qué se usa?

Es el núcleo estructural del sistema. Gestiona todos los datos que tienen relaciones bien definidas entre sí: un usuario se postula a una oferta publicada por una empresa, tiene habilidades, pertenece a grupos y tiene historial de experiencia y educación. También maneja el sistema de autenticación y los roles (candidato / reclutador).

Cada vez que un reclutador crea una oferta, cuando un candidato se postula, cuando el estado de una postulación cambia (enviada → en revisión → aceptada / rechazada), o cuando se registra un nuevo usuario, todo pasa por PostgreSQL.

¿Por qué PostgreSQL?

  • Los datos tienen estructura fija y conocida: columnas claras, tipos definidos, claves foráneas entre tablas.
  • Las relaciones entre entidades son complejas: un empleo tiene habilidades requeridas, una postulación pertenece a un usuario y a un empleo, una habilidad puede estar en múltiples usuarios y empleos.
  • Necesitamos transacciones ACID: si una postulación falla a mitad de camino, los datos no pueden quedar en estado inconsistente.
  • El historial de estados de una postulación requiere integridad garantizada — no podemos perder ese registro.
  • SQL permite consultas complejas con joins entre múltiples tablas, ideales para listar empleos con sus habilidades y empresa, o postulaciones con los datos del candidato.

¿Qué aporta Supabase?

  • Supabase es un Backend-as-a-Service que corre sobre PostgreSQL y agrega: autenticación (login, tokens JWT), API REST automática (PostgREST) y Row Level Security.
  • Se usan dos clientes en el backend: el cliente anon para operaciones públicas y el cliente service (con permisos de administrador) para operaciones que bypassean los permisos de fila.
  • Permite buscar empleos por título, empresa o habilidad con una sola consulta combinando ilike y or.
  • El soft-delete de postulaciones (cambiar estado a cancelled en lugar de borrar) se implementa aquí para conservar el historial de auditoría.

Base de datos 2

Documental · NoSQL MongoDB

¿Qué datos guarda?

publicaciones del feed publicaciones en grupos comentarios likes

¿Para qué se usa?

Gestiona todo el contenido social que los usuarios generan: lo que publican en el feed general, lo que publican dentro de grupos, los comentarios que dejan en las publicaciones y los likes. Cada publicación se almacena como un documento JSON completo que contiene el texto, el autor, la fecha, las etiquetas y el contador de interacciones.

Cuando un usuario abre el feed, se consultan los posts ordenados por fecha. Cuando comenta, se inserta un nuevo documento en la colección de comentarios vinculado al post por su post_id. Cuando da like, se actualiza el contador directamente en el documento del post.

¿Por qué MongoDB?

  • Una publicación puede tener texto solamente, o texto con imagen, o pertenecer a un grupo, o tener etiquetas — su estructura varía. El modelo documental acepta documentos con campos distintos sin necesidad de definir un esquema rígido.
  • El contenido social evoluciona seguido: si queremos agregar un nuevo campo (por ejemplo, un "tipo de reacción" distinto al like), se agrega en nuevos documentos sin tocar los existentes ni hacer una migración de base de datos.
  • Las consultas del feed son simples: buscar por autor, ordenar por fecha, filtrar por etiqueta. No hacen falta joins complejos.
  • MongoDB tiene índices compuestos sobre author_id, created_at y tags que hacen eficientes estas consultas.
  • El patrón de relaciones embebidas vs. enlazadas (visto en la materia) se aplica: los likes son un contador embebido en el post, pero los comentarios son documentos separados enlazados por post_id, ya que su número puede crecer indefinidamente.

¿Por qué no PostgreSQL para esto?

  • Guardar posts en SQL obligaría a definir columnas fijas para todos los campos posibles, muchas de las cuales quedarían vacías según el tipo de publicación.
  • Los comentarios anidados y los likes representados como array en un documento JSON son más naturales que tablas relacionales con joins.
  • El contenido social tiene más escrituras y lecturas simples que consultas analíticas complejas — justamente el caso de uso donde MongoDB brilla frente a SQL.
  • MongoDB escala horizontalmente (sharding) si el volumen de posts crece — algo más complejo de lograr con PostgreSQL.

Base de datos 3

Grafos · NoSQL Neo4j

¿Qué datos guarda?

Nodos

:User :Job :Skill

Cada nodo tiene propiedades: user_id, job_id o el nombre de la habilidad.

Relaciones

CONNECTED_WITH PENDING_REQUEST HAS_SKILL REQUIRES_SKILL

¿Para qué se usa?

Red de conexiones: cuando un usuario manda una solicitud de conexión se crea la relación PENDING_REQUEST. Al aceptarla, se reemplaza por CONNECTED_WITH. Esto modela la red de contactos de la misma forma en que LinkedIn la tiene: un grafo de personas conectadas entre sí.

Sugerencias de personas: el algoritmo busca en Neo4j usuarios que estén conectados a los contactos del usuario actual (amigos de amigos), que tengan habilidades en común, y que aún no estén conectados con él. Este tipo de recorrido es trivial en un grafo y muy costoso con joins en SQL.

Job match: dado que los empleos tienen relaciones REQUIRES_SKILL y los usuarios tienen HAS_SKILL, Neo4j puede encontrar en una sola consulta los empleos que mejor se ajustan al perfil del usuario — sin hacer cruce de tablas en SQL.

¿Por qué Neo4j?

  • En una red social, las relaciones son el dato más importante. "¿Quién conoce a quién?" es una pregunta de grafo, no de tabla.
  • Con SQL, calcular sugerencias de conexión requeriría múltiples joins anidados que se vuelven muy lentos a medida que la red crece. En Neo4j es un simple recorrido de dos saltos.
  • Neo4j usa almacenamiento nativo de grafos: cada nodo tiene punteros directos a sus relaciones, lo que hace los recorridos extremadamente rápidos.
  • El lenguaje Cypher permite expresar consultas de red de forma muy natural: (usuario)-[:CONNECTED_WITH]->(contacto)-[:CONNECTED_WITH]->(sugerencia).
  • La materia señala que LinkedIn usa bases de grafos exactamente para esto.

Detalles de implementación

  • Al crear un usuario en Supabase, automáticamente se crea su nodo :User en Neo4j — los dos sistemas se mantienen sincronizados.
  • Al publicar un empleo, se crea el nodo :Job con sus relaciones REQUIRES_SKILL.
  • Se usan restricciones UNIQUE en user_id y job_id para garantizar que no haya nodos duplicados.
  • Hay un índice sobre Skill.name para encontrar habilidades rápidamente.
  • Si Neo4j no está disponible, el sistema sigue funcionando — las sugerencias simplemente no se muestran (diseño tolerante a fallos).

Base de datos 4

Tabular · NoSQL AstraDB — Apache Cassandra

¿Qué datos guarda?

mensajes directos entre usuarios conversaciones notificaciones

¿Para qué se usa?

Mensajería: cuando un usuario envía un mensaje a otro, ese mensaje se escribe en Cassandra dentro de la conversación correspondiente. Al abrir una conversación se leen todos los mensajes ordenados por timestamp. La clave de partición es el conversation_id, lo que garantiza que todos los mensajes de una misma conversación estén en el mismo nodo del clúster y se puedan leer juntos sin acceder a múltiples servidores.

Notificaciones: cada acción relevante del sistema (nueva solicitud de conexión, postulación aceptada, nuevo comentario en un post) genera una notificación que se escribe en Cassandra. Al cargar la pantalla del usuario se leen sus notificaciones recientes ordenadas por fecha.

¿Por qué Cassandra?

  • Los mensajes y notificaciones se escriben constantemente y en gran volumen. Cassandra está diseñada para esto: sus escrituras son muy baratas (van primero al Commit Log en memoria, luego al disco), mucho más que en una base relacional.
  • Es una base sin punto único de falla: todos los nodos son iguales, no hay nodo "master". Si cae un nodo, los demás siguen respondiendo.
  • Los mensajes son datos append-only (siempre se agregan, casi nunca se editan), que es exactamente el patrón para el que Cassandra está optimizada.
  • Para las notificaciones es aceptable la consistencia eventual: si una notificación aparece un segundo después de que ocurrió el evento, no es un problema crítico. Cassandra prioriza disponibilidad sobre consistencia estricta.
  • Facebook desarrolló Cassandra en 2007 para gestionar búsquedas en bandejas de entrada a escala masiva — el mismo caso de uso que aplicamos acá.

Detalles de implementación

  • Se usa AstraDB, que es Cassandra administrado en la nube (DataStax). No hay que mantener un clúster propio — se accede con un token y la URL del endpoint.
  • El SDK es astrapy (versión 2.0), la librería oficial de Python para AstraDB.
  • Las notificaciones pueden tener TTL (tiempo de vida): Cassandra las marca como expiradas automáticamente sin necesidad de borrarlas manualmente.
  • El sistema fue diseñado con degradación elegante: si AstraDB no está disponible en algún momento, el backend no cae — simplemente los mensajes y notificaciones no se sirven hasta que el servicio se recupere.
  • La clave de partición de los mensajes es el conversation_id, garantizando que todos los mensajes de una conversación vivan en el mismo nodo.
