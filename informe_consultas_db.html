1. Supabase — PostgreSQL

Cómo conectamos

La conexión se establece en backend/db.py usando el SDK oficial supabase-py. Se crean dos clientes al arrancar el servidor, que quedan disponibles para toda la aplicación:

# backend/db.py
from supabase import create_client

# Cliente público (operaciones normales con permisos de fila)
supabase = create_client(
    os.environ["SUPABASE_URL"],
    os.environ["SUPABASE_ANON_KEY"],
)

# Cliente administrador (bypass RLS — solo para el backend)
supabase_admin = create_client(
    os.environ["SUPABASE_URL"],
    os.environ["SUPABASE_SERVICE_KEY"],
)

Además, al arrancar el servidor se abre una conexión directa con psycopg2 para ejecutar el DDL de creación de tablas (init_db.py), usando la variable DATABASE_URL.

Dónde se ejecutan las queries

En el motor PostgreSQL (servidor Supabase)

  • Todas las consultas SQL (SELECT, INSERT, UPDATE, DELETE) se ejecutan en el servidor de Supabase en la nube.
  • Los triggers set_updated_at() se disparan automáticamente en el motor cuando se actualiza una fila en users, jobs o applications.
  • Los índices los usa el motor para planificar y optimizar cada consulta.
  • Las restricciones (UNIQUE, FK, CHECK) las valida el motor antes de confirmar la escritura.

En el backend Flask (Python)

  • El backend construye la consulta usando la API fluida del SDK y la envía al servidor.
  • El filtrado adicional por job_ids (cruce de habilidades y empresa) se arma en Python antes de enviar la query final.
  • La hidratación de datos (enriquecer posts de MongoDB con datos de autor de Supabase) también ocurre en el backend.

Cómo se consulta — API fluida del SDK

No se escribe SQL a mano. El SDK de Supabase expone una API encadenable que se traduce a queries PostgreSQL del lado del servidor:

# Buscar empleos activos con empresa y habilidades (JOIN automático)
supabase_admin.table("jobs")
  .select("*, companies(name), job_skill(skills(skill_id, name, type))")
  .eq("is_active", True)
  .order("created_at", desc=True)
  .execute()
# Insertar postulación
supabase_admin.table("applications").insert({
    "application_id": str(uuid.uuid4()),
    "job_id": job_id,
    "user_id": user_id,
    "status": "submitted",
}).execute()
# Actualizar estado de postulación
supabase_admin.table("applications")
  .update({"status": status})
  .eq("application_id", app_id)
  .execute()
# Búsqueda de empleos por texto (title, location, empresa, habilidad)
base.or_(f"title.ilike.%{q}%,location.ilike.%{q}%,job_id.in.({ids})").execute()

Estructuras de datos — esquema relacional

Identidad y roles

  • users: user_id (PK), email, password_hash, name, surname, dni, profile_photo_url
  • roles: role_id, name → "candidato" / "poster"
  • user_roles: (user_id, role_id) PK compuesta

Perfil profesional

  • companies: company_id, name, slogan, num_employees, location
  • work experience: we_id, user_id FK, company_id FK, title, from_date, end_date, is_current
  • education: edu_id, user_id FK, title, field, institution, from_date
  • skills: skill_id, name UNIQUE, type
  • user_skill: (user_id, skill_id) PK + level

Empleos y postulaciones

  • jobs: job_id, poster_user_id FK, company_id FK, title, description, location, modality, shift, employment_type, is_active
  • job_skill: (job_id, skill_id) PK
  • applications: application_id PK, user_id FK, job_id FK, status, UNIQUE(user_id, job_id)
  • application_status_history: history_id, application_id FK, status, changed_at
  • groups + group_members
Triggers automáticos: PostgreSQL ejecuta set_updated_at() antes de cada UPDATE en users, jobs y applications, actualizando la columna updated_at sin que el backend lo tenga que hacer manualmente.

2. MongoDB

Cómo conectamos

La conexión se maneja en backend/db_mongo.py con el patrón singleton: se crea el cliente la primera vez que se pide y se reutiliza en todas las peticiones. Al conectar, se crean automáticamente las colecciones e índices si no existen.

# backend/db_mongo.py
from pymongo import MongoClient, ASCENDING, DESCENDING

_client = None
_db     = None

def get_db():
    global _client, _db
    if _db is None:
        _client = MongoClient(
            os.environ["MONGO_URI"],
            serverSelectionTimeoutMS=8000,
            tls=True,
        )
        _db = _client[os.environ.get("MONGO_DB", "linkpro")]
        _init_collections(_db)   # crea colecciones e índices
    return _db

Dónde se ejecutan las queries

Todo se ejecuta en el servidor de MongoDB Atlas. El backend solo arma el pipeline o el filtro y lo envía. En particular, el Aggregation Pipeline (que hace el lookup de likes y comments, cuenta, filtra) corre completamente dentro del motor de MongoDB — no en Python.

El único procesamiento que ocurre en el backend es la hidratación: después de traer los posts de MongoDB, se consulta a Supabase para obtener los datos actualizados del autor (nombre, foto) y se combinan en Python antes de enviar la respuesta al frontend.

Cómo se consulta

Se usa pymongo directamente. Las consultas simples usan find / insert_one / delete_one. Para el feed se usa un Aggregation Pipeline que hace joins internos entre colecciones:

# Obtener feed con likes y comentarios en una sola query
pipeline = [
    {"$match": {"group_id": {"$exists": False}}},  # solo posts del feed general
    {"$sort": {"created_at": -1}},
    {"$limit": 50},
    {"$lookup": {                          # JOIN con colección likes
        "from": "likes",
        "localField": "_id",
        "foreignField": "post_id",
        "as": "likes_list",
    }},
    {"$lookup": {                          # JOIN con colección comments
        "from": "comments",
        "localField": "_id",
        "foreignField": "post_id",
        "as": "comments_list",
        "pipeline": [{"$sort": {"created_at": 1}}],
    }},
    {"$addFields": {                       # campos calculados
        "likes_count":    {"$size": "$likes_list"},
        "comments_count": {"$size": "$comments_list"},
        "user_liked":     {"$in": [user_id, "$likes_list.user_id"]},
    }},
]
docs = list(db.posts.aggregate(pipeline))
# Insertar un post
db.posts.insert_one({
    "user_id": user_id, "content": content,
    "tags": ["python", "nosql"],
    "created_at": datetime.now(timezone.utc),
})

# Toggle like (insert o delete según si ya existía)
existing = db.likes.find_one({"post_id": oid, "user_id": user_id})
if existing:
    db.likes.delete_one({"_id": existing["_id"]})
else:
    db.likes.insert_one({"post_id": oid, "user_id": user_id, ...})

# Comentarios de un post ordenados por fecha
db.comments.find({"post_id": oid}).sort("created_at", 1)

Estructuras de datos — documentos JSON por colección

Colección: posts

  • _id: ObjectId (auto)
  • user_id: string
  • author_name / author_surname
  • author_photo_url
  • content: string
  • media: url o null
  • tags: array de strings
  • group_id: string (opcional)
  • created_at / updated_at: datetime

Colección: likes

  • _id: ObjectId
  • post_id: ObjectId (ref a posts)
  • user_id: string
  • created_at: datetime

Índice único sobre (post_id, user_id) → garantiza un like por usuario por post.

Colección: comments

  • _id: ObjectId
  • post_id: ObjectId (ref a posts)
  • user_id: string
  • author_name / author_surname
  • text: string
  • created_at: datetime

Índices creados

  • posts.created_at DESC
  • posts.(user_id, created_at DESC)
  • posts.tags
  • posts.(group_id, created_at DESC) sparse
  • likes.(post_id, user_id) unique
  • likes.user_id
  • comments.(post_id, created_at ASC)
  • comments.user_id

3. Neo4j

Cómo conectamos

La conexión se maneja en backend/db_neo4j.py. Se usa el driver oficial de Python para Neo4j con patrón singleton. Al conectar por primera vez, se crean automáticamente las restricciones e índices en el grafo.

# backend/db_neo4j.py
from neo4j import GraphDatabase

_driver = None

def get_driver():
    global _driver
    if _driver is None:
        _driver = GraphDatabase.driver(
            os.environ["NEO4J_URI"],
            auth=(os.environ["NEO4J_USER"], os.environ["NEO4J_PASSWORD"])
        )
        with _driver.session() as s:
            _init_indexes(s)   # crea constraints e índices
    return _driver

def _init_indexes(session):
    session.run("CREATE CONSTRAINT user_id_unique IF NOT EXISTS FOR (u:User) REQUIRE u.user_id IS UNIQUE")
    session.run("CREATE CONSTRAINT job_id_unique  IF NOT EXISTS FOR (j:Job)  REQUIRE j.job_id  IS UNIQUE")
    session.run("CREATE INDEX skill_name_idx    IF NOT EXISTS FOR (s:Skill) ON (s.name)")

Dónde se ejecutan las queries

Las queries Cypher se ejecutan en el servidor de Neo4j AuraDB (nube). El backend envía la query como texto con parámetros. El motor de Neo4j planifica y recorre el grafo de forma nativa — los punteros directos entre nodos hacen que los recorridos de múltiples saltos sean muy rápidos, sin necesidad de escanear tablas.

Cómo se consulta — lenguaje Cypher

Todo se consulta con Cypher. El backend abre una sesión, ejecuta la query con parámetros y lee los resultados. Para escrituras se usa execute_write() que garantiza que la transacción se reintente si hay conflicto.

# Crear/actualizar nodo de usuario (MERGE = insert si no existe, update si existe)
session.run(
    "MERGE (u:User {user_id: $uid}) "
    "SET u.name = $name, u.surname = $surname, u.photo_url = $photo",
    uid=user_id, name=name, surname=surname, photo=photo_url
)
# Agregar habilidad al usuario (crea el nodo Skill si no existe, crea la relación)
session.run(
    "MERGE (s:Skill {name: $name}) WITH s "
    "MATCH (u:User {user_id: $uid}) "
    "MERGE (u)-[r:HAS_SKILL]->(s) SET r.level = $level",
    name=skill_name, uid=user_id, level="Intermedio"
)
# Sugerencias de conexión: amigos de amigos ordenados por mutuos en común
session.run(
    "MATCH (u:User {user_id: $uid})-[:CONNECTED_WITH]-(mid:User)-[:CONNECTED_WITH]-(s:User) "
    "WHERE s.user_id <> $uid AND NOT s.user_id IN $excluded "
    "WITH s, count(DISTINCT mid) AS mutuals "
    "ORDER BY mutuals DESC LIMIT 10 "
    "RETURN s, mutuals",
    uid=user_id, excluded=excluded_ids
)
# Job match: empleos que requieren las habilidades del usuario
session.run(
    "MATCH (u:User {user_id: $uid})-[:HAS_SKILL]->(s:Skill)<-[:REQUIRES_SKILL]-(j:Job) "
    "RETURN j.job_id AS job_id, collect(s.name) AS matching_skills, count(s) AS match_count "
    "ORDER BY match_count DESC LIMIT 20",
    uid=user_id
)
# Verificar si dos usuarios están conectados (antes de permitir mensajes)
session.run(
    "MATCH (a:User {user_id: $from})-[:CONNECTED_WITH]-(b:User {user_id: $to}) RETURN a LIMIT 1",
    {"from": from_id, "to": to_id}
).single()

Estructuras de datos — nodos y relaciones del grafo

Nodos

  • :User — user_id, name, surname, photo_url
  • :Job — job_id
  • :Skill — name

Restricción UNIQUE sobre User.user_id y Job.job_id. Índice sobre Skill.name.

Relaciones

  • CONNECTED_WITH — bidireccional entre dos :User
  • PENDING_REQUEST — dirigida: solicitante → receptor. Propiedad: created_at
  • HAS_SKILL — :User → :Skill. Propiedad: level
  • REQUIRES_SKILL — :Job → :Skill

Al aceptar una conexión, PENDING_REQUEST se borra y se crean dos relaciones CONNECTED_WITH (una en cada dirección).

4. AstraDB — Apache Cassandra

Cómo conectamos

La conexión se maneja en backend/db_cassandra.py usando astrapy, el SDK oficial de DataStax para la Data API de AstraDB. No se usa CQL directo sino una API tipo documento (similar a MongoDB).

# backend/db_cassandra.py
from astrapy import DataAPIClient

_db = None

def get_db():
    global _db
    if _db is None:
        client = DataAPIClient(os.environ["ASTRA_TOKEN"])
        _db = client.get_database(os.environ["ASTRA_ENDPOINT"])
    return _db

def get_collection(name: str):
    # Crea la colección si no existe, devuelve el handle
    db = get_db()
    existing = {c.name for c in db.list_collections()}
    if name not in existing:
        return db.create_collection(name)
    return db.get_collection(name)
¿Por qué astrapy y no CQL? AstraDB expone una Data API HTTP que permite operar con documentos JSON usando una sintaxis similar a MongoDB. No es necesario conectarse por el protocolo nativo de Cassandra ni escribir CQL para operaciones básicas. Esto simplifica mucho la integración desde un backend Python.

Dónde se ejecutan las queries

En AstraDB (Cassandra en la nube)

  • Todas las operaciones de lectura/escritura se ejecutan en el clúster de Cassandra administrado por DataStax.
  • La Data API traduce internamente las operaciones de documento a CQL y las ejecuta en el motor de Cassandra.
  • La replicación y distribución entre nodos las maneja Cassandra de forma automática y transparente.

En el backend Flask

  • El conversation_id se calcula en Python: se ordenan los dos user_id alfabéticamente y se concatenan con _, garantizando que siempre sea el mismo sin importar quién inició la conversación.
  • El backend también consulta a Supabase para obtener el nombre y foto de los participantes de cada conversación, y los combina con los datos de AstraDB antes de responder.

Cómo se consulta

La API de astrapy tiene una sintaxis muy similar a pymongo: find, insert_one, update_one, update_many, delete_one.

# Insertar una notificación
get_collection("notifications").insert_one({
    "_id":        str(uuid.uuid4()),
    "user_id":   user_id,
    "type":      "connection_request",
    "body":      "Juan Pérez te envió una solicitud de conexión.",
    "ref_id":    sender_id,
    "is_read":   False,
    "created_at": datetime.now(timezone.utc).isoformat(),
})
# Leer notificaciones de un usuario, ordenadas por fecha
list(_coll().find(
    {"user_id": user_id},
    sort={"created_at": -1},
    limit=50
))
# Enviar un mensaje (inserta mensaje y crea/actualiza conversación)
_msgs().insert_one({
    "_id":             str(uuid.uuid4()),
    "conversation_id": convo_id,       # "userId1_userId2" (ordenado)
    "sender_id":       from_id,
    "receiver_id":     to_id,
    "body":            body,
    "sent_at":         now,
    "is_read":         False,
})

# Actualizar timestamp de la conversación
_convos().update_one(
    {"_id": convo_id},
    {"$set": {"last_message": body, "last_sent_at": now}}
)

# Marcar todos los mensajes de una conversación como leídos
_msgs().update_many(
    {"conversation_id": convo_id, "receiver_id": reader_id, "is_read": False},
    {"$set": {"is_read": True}}
)

Estructuras de datos — colecciones en AstraDB

Colección: messages

  • _id: UUID (generado en Python)
  • conversation_id: string (uid1_uid2)
  • sender_id: string
  • receiver_id: string
  • body: string
  • sent_at: ISO datetime
  • is_read: boolean
  • edited: boolean (opcional)

Colección: conversations

  • _id: "uid1_uid2" (determinístico)
  • participants: [uid1, uid2]
  • last_message: string
  • last_sent_at: ISO datetime
  • last_sender_id: string
  • names: {uid: "Nombre Apellido"}
  • photos: {uid: url}

Colección: notifications

  • _id: UUID
  • user_id: string
  • type: "like" | "comment" | "connection_request" | "application_update"
  • body: texto de la notificación
  • ref_id: ID del objeto referenciado (post, job, etc.)
  • is_read: boolean
  • created_at: ISO datetime
